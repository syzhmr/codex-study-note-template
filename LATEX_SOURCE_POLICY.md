# LaTeX ソース可読性方針

このファイルは，LaTeX ソースを人間と AI の両方が読みやすい形に保つための補助方針である。
数学本文の内容，証明，引用，用語に関する判断は，必ず `NOTE_POLICY.md`，`TERM_DICTIONARY.md`，`HANDWRITTEN_INDEX.md` を優先する。

## 優先順位

1. ユーザーの明示的な指示を最優先する。
2. 数学本文については `NOTE_POLICY.md`，用語については `TERM_DICTIONARY.md`，ノート対応については `HANDWRITTEN_INDEX.md` を優先する。
3. 既存のファイル構成，既存マクロ，既存のビルド方法を優先する。
4. この可読性方針は，意味を変えない整形・構造化・命名・差分管理のために使う。

## ソース整形

- 本文は原則として 1 文 1 行にする。
- 長すぎる行は，文，式，条件，理由などのまとまりで分ける。
- 段落途中で意味のない改行を大量に入れない。
- 空行は，段落，大きな構造，意味の切れ目を示すために使う。
- `\begin{...}` と `\end{...}` の対応が見えるように揃える。
- 数式環境，図式環境，表環境の内部は，自動整形で不用意に触らない。

## ファイル構成

- `main_final.tex` は文書全体の構造を管理する。
- `preemble.tex` はプリアンブルとして扱う。
- `final/chapter01.tex` などの chapter ファイルは，章・節見出しと小節ファイルへの `\input` を管理する。
- 各小小節の本文は，`final/chapter01/1-1-1.tex` のように小節単位のファイルに置く。
- subsection 直後の導入文は，冗長な `intro` ファイルを作らず，親の chapter ファイルに残してよい。

## マクロとプリアンブル

- 既存マクロの意味を変更しない。
- 既存マクロと似た役割の新規マクロを不用意に増やさない。
- パッケージ読み込み順，LaTeX エンジン，文書クラスは指示なく変更しない。
- 使われていないパッケージの削除は，コンパイル確認と影響確認ができる場合だけ行う。

## 数式と証明

- 数式の意味，仮定，結論，番号の有無を変えない。
- 番号付き数式を番号なしに変更しない。
- 複数行数式は，等号や関係演算子の位置が読みやすくなるように整える。
- 証明の論理構造を整形だけを理由に変えない。
- 長い証明を段落や局所的な主張に分ける場合は，数学的順序を変えず，`NOTE_POLICY.md` の証明方針に従う。

## 参考文献

- 引用キー，引用位置，参考文献情報を推測で変更しない。
- DOI，URL，出版年，著者名などの書誌情報は，信頼できる情報源で確認できる場合だけ補う。
- 参考文献に基づく補足は，`references/notes/` と `references/REFERENCE_INDEX.md` に根拠を残す。

## ビルド

- 既存の `latexmkrc` と現在のビルド方法を優先する。
- TeX 本文を変更した場合は，原則として `latexmk main_final.tex` でビルドする。
- ビルド成功時に `main_final.pdf` と `main_final.synctex.gz` が生成されることを確認する。
- 生成物や補助ファイルは，指示がない限り編集対象にしない。

## 報告

作業後は，変更したファイル，主な変更内容，実行したビルド・検査コマンド，コンパイル結果，未解決事項を簡潔に報告する。

## Git

このテンプレートを用いるプロジェクトでは Git 利用を必須とする。
ソース整形や本文改訂の前に `git status --short` を確認し，作業後に `git diff` を確認する。
内容変更と整形変更は commit を分ける。
生成物，PDF，参考文献 PDF，ユーザー作成ノート，抽出画像は原則として Git に入れない。
