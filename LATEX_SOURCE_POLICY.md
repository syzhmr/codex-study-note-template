# LaTeX ソース可読性方針

このファイルは，LaTeX ソースを人間と AI の両方が読みやすい形に保つための補助方針である。
数学本文の内容，証明，引用，用語に関する判断は，必ず `WORKFLOW_STEPS.md`，`NOTE_POLICY.md`，`TERM_DICTIONARY.md`，`HANDWRITTEN_INDEX.md` を優先する。

## 優先順位

1. `WORKFLOW_STEPS.md` で作業型を確認する。
2. ユーザーの明示的な指示を最優先する。
3. 数学本文については `NOTE_POLICY.md`，用語については `TERM_DICTIONARY.md`，ノート対応については `HANDWRITTEN_INDEX.md`，証明補完については `HANDWRITTEN_PROOF_POLICY.md` を優先する。
4. 既存のファイル構成，既存マクロ，既存のビルド方法を優先する。
5. この可読性方針は，意味を変えない整形・構造化・命名・差分管理のために使う。

この方針と数学的内容の正確性が衝突する場合は，数学的内容を変えないことを優先する。

## ソース整形

- 本文は原則として 1 文 1 行にする。
- 長すぎる行は，文，式，条件，理由などのまとまりで分ける。
- 段落途中で意味のない改行を大量に入れない。
- 空行は，段落，大きな構造，意味の切れ目を示すために使う。
- 連続する空行は原則 1 行にまとめる。
- `\begin{...}` と `\end{...}` の対応が見えるように揃える。
- 環境，箇条書き，複数行数式は，既存の書き方に合わせて読みやすくインデントする。
- 数式環境，図式環境，表環境の内部は，自動整形で不用意に触らない。
- 長い URL を含む `\href` や `\url` は，リンク保持を優先して無理に分割しない。

## ファイル構成

- `main_final.tex` は文書全体の構造を管理する。
- `preemble.tex` はプリアンブルとして扱う。ファイル名は現状を尊重し，指示なく変更しない。
- `final/chapter01.tex` などの chapter ファイルは，章・節見出しと小節ファイルへの `\input` を管理する。
- 各小小節の本文は，`final/chapter01/1-1-1.tex` のように小節単位のファイルに置く。
- subsection 直後の導入文は，冗長な `intro` ファイルを作らず，親の chapter ファイルに残してよい。
- 完成 PDF 末尾の参考文献欄は `final/references.tex` に置く。

## マクロとプリアンブル

- 既存マクロの意味を変更しない。
- 既存マクロと似た役割の新規マクロを不用意に増やさない。
- 同じ記号・書式が何度も現れ，既存方針と衝突しない場合だけ，意味に基づくマクロ化を検討する。
- パッケージ読み込み順，LaTeX エンジン，文書クラスは指示なく変更しない。
- 使われていないパッケージの削除は，コンパイル確認と影響確認ができる場合だけ行う。

## 定理環境の装飾

- 定理，定義，命題，補題，系，主張の装飾は，数学的構造を見えやすくするために使う。
- この装飾は「トゥー多様体(裳華房)風の定理環境」と呼ぶ。「Mathlog風」とは呼ばない。
- 装飾の参考例として，Mathlog「〖LaTeX〗「トゥー多様体」風の装飾」<https://mathlog.info/articles/AvtyQT4nCGaXFTEVVTpC> を参照する。
- 参考記事の方針は，`tcolorbox` による左側の縦線，定理の強い強調，命題・補題・系の控えめな強調，定義見出しのボックス化として取り入れる。ただし，サンプルをそのままコピーせず，このテンプレートの環境名とビルド方法に合わせる。
- 見出しを枠で囲むのは定義だけにする。命題，定理，補題，系，主張の見出しは太字とし，番号や見出しを定義用のバッジで囲まない。
- 色は本文読解を助ける範囲に抑える。
- 装飾変更は本文改訂とは別の commit 単位にし，`main_final.tex` のビルドと PDF の目視確認を行う。

## ラベルと参照

- 既存ラベルは，不用意に変更しない。
- 新しいラベルを追加する場合は，節は `sec:`，定義は `dfn:`，定理は `thm:`，補題は `lem:`，主張は `claim:`，命題は `prop:`，系は `cor:` を用いる。
- 数式ラベルを追加する場合は `eq:` を用いる。
- 意味の薄いラベル名は避ける。
- 参照表記は既存マクロと既存の書き方に合わせる。
- 参照先が存在しないラベルを作らない。

## 数式と証明

- 数式の意味，仮定，結論，番号の有無を変えない。
- 番号付き数式を番号なしに変更しない。番号なし数式を番号付きに変更しない。
- 複数行数式は，等号や関係演算子の位置が読みやすくなるように整える。
- 長い数式は，意味のまとまりごとに改行する。
- 証明の論理構造を整形だけを理由に変えない。
- 長い証明を段落や証明段階に分ける場合は，数学的順序を変えず，`NOTE_POLICY.md` と `HANDWRITTEN_PROOF_POLICY.md` に従う。
- 作業記録で `Step` と呼ぶものは，完成版に出す見出し名ではなく，主テキストと手書きノートから読み取る証明の大きな役割分担である。
- `claim` や局所的な `lem` は，各証明段階の中で使う具体的な確認，計算，帰着，同型性，単射性，全射性などを示すために置き，証明段階そのものの代わりにしない。
- 証明内で補助 `claim` や局所的な `lem` を追加する前に，同じ知識が既に本文，注意，定義，命題，補題，定理，主張，例で示されていないか確認する。
- 証明内で補助 `claim` または局所的な `lem` を使う場合は，環境の直前に「まず次の主張を示す。」「次の補題を示す。」などの文を置く。
- 証明内の補助 `claim` は直前の証明文脈に依存してよい。
- 証明内の局所的な `lem` は，命題として独立して読めるように，対象，仮定，結論を環境内に書く。
- 補助主張または局所補題の証明が終わる箇所には，その局所証明を閉じる最後の文の直後に `\localqed` を置き，音符で局所的な証明終了を示す。音符だけを独立行に置かない。

## 参考文献

- 引用キー，引用位置，参考文献情報を推測で変更しない。
- DOI，URL，出版年，著者名などの書誌情報は，信頼できる情報源で確認できる場合だけ補う。
- 本文中の注釈で実際に引用箇所を明示する場合は，節番号・定理番号・ページ番号・URL をできるだけ具体的に書く。
- 完成 PDF 末尾の参考文献一覧では，ページ番号や定理番号ではなく書誌情報を中心にまとめる。
- 参考文献に基づく補足は，`references/notes/` と `references/REFERENCE_INDEX.md` に根拠を残す。
- 新しい参考文献が増えた場合は，追加時点で `final/references.tex` に直行させない。先に台帳化と使用箇所の確認を行う。

## ビルド

- 既存の `latexmkrc` と現在のビルド方法を優先する。
- TeX 本文を変更した場合は，原則として `latexmk -g -dvi -interaction=nonstopmode main_final.tex` でビルドする。
- ビルド成功時に `main_final.pdf` と `main_final.synctex.gz` が生成されることを確認する。
- 生成物や補助ファイルは，指示がない限り編集対象にしない。

## Git

このテンプレートを用いるプロジェクトでは Git 利用を必須とする。
ソース整形や本文改訂の前に `git status --short --branch` を確認し，作業後に `git diff` を確認する。
内容変更と整形変更は commit を分ける。
生成物，主テキスト PDF，参考文献 PDF，ユーザー作成ノート，抽出画像は原則として Git に入れない。
例外的に PDF を Git 管理する場合は，権利と容量を確認し，100MB を超える PDF は分割して各ファイルを 100MB 未満にする。

## 報告

作業後は，変更したファイル，主な変更内容，実行したビルド・検査コマンド，コンパイル結果，未解決事項を簡潔に報告する。
