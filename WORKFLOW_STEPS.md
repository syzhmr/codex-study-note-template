# 再現可能作業手順

このファイルを Codex 作業の入口にする。
目的は，モデルや実行環境が変わっても，同じ順序で確認し，同じ条件で止まり，同じ形で報告できるようにすることである。

## 方針改訂の共通前提

1. 手順化の目的は，処理を軽くすることではない。
2. 目的は，暗黙の文脈を減らし，将来どのモデルで作業しても同じ判断順になるようにすることである。
3. 方針を改訂するときは，まず既存の関連方針全体を横断的に読み直す。
4. 読み直しでは，重複，長文化，役割分担の不明瞭さ，判断順の曖昧さ，停止条件の不足を洗い出す。
5. 入口手順に置くべき判断と，各方針ファイルに残すべき詳細を分ける。
6. 長い説明は，小さな判断手順，停止条件，報告項目へ分解する。

## 0. 入口判定

1. `git status --short --branch` を確認する。
2. ユーザーの依頼を次の型に分ける。
   - 方針変更
   - 本文改訂
   - LaTeX 整形
   - 用語確認
   - 参考文献確認
   - 手書きノート確認
   - Git 操作
3. 型に応じて読むファイルを絞る。
4. 依頼範囲が広い場合でも，本文改訂は原則として `subsubsection` 単位に切る。対象に `subsubsection` がない場合だけ，最小の節単位に切る。
5. 未コミット変更がある場合は，変更元と作業範囲を確認し，無関係な変更を壊さない。

## 1. 読むファイル

### 常に読む

1. `WORKFLOW_STEPS.md`
2. `PROJECT_PROFILE.md`
3. `GIT_WORKFLOW.md`

### 方針変更で読む

1. `README.md`
2. `NOTE_POLICY.md`
3. `LATEX_SOURCE_POLICY.md`
4. `HANDWRITTEN_INDEX.md`
5. `HANDWRITTEN_PROOF_POLICY.md`
6. `TERM_DICTIONARY.md`
7. `references/README.md`
8. `references/notes/README.md`
9. `references/REFERENCE_INDEX.md`
10. 参考文献一覧に関わる場合は `references/notes/final_bibliography_style.md`
11. 手書きノートの略記・色付き矢印に関わる場合は `references/notes/handwritten_reference_aliases.md`
12. 修正文候補に関わる場合は `REVISION_CANDIDATES.md`

### 本文改訂で読む

1. 対象範囲の TeX。
2. `NOTE_POLICY.md`。
3. `HANDWRITTEN_INDEX.md` の対象行。
4. 証明補完を伴う場合は `HANDWRITTEN_PROOF_POLICY.md`。
5. `TERM_DICTIONARY.md` の対象語。
6. 手書きノートの参考文献候補，略記，色付き矢印を使う場合は `references/notes/handwritten_reference_aliases.md`。
7. `references/notes/README.md` の探す順序に従って選んだ関連メモ。

### LaTeX 整形で読む

1. `LATEX_SOURCE_POLICY.md`
2. 対象 TeX
3. 意味が変わる可能性がある場合だけ `NOTE_POLICY.md`

### 参考文献確認で読む

1. `references/notes/README.md`
2. `references/REFERENCE_INDEX.md`
3. `references/notes/` の関連メモ
4. 必要な PDF または公開 URL
5. 完成 PDF 末尾の参考文献一覧に関わる場合は `references/notes/final_bibliography_style.md`

新しい参考文献を追加する場合は，追加時点で末尾一覧に入れない。
台帳化，使用箇所確認，本文反映，末尾一覧への登録の順に処理する。

## 2. 本文改訂の最小単位

1. 対象を 1 つの `subsubsection` に固定する。
2. 章全体を先に一括通読して薄く直さない。章全体の依頼では，最初に `subsubsection` 一覧と順序だけを作り，各 `subsubsection` の中で主テキスト，ユーザー作成ノート，参考文献，既存 TeX を通読する。
3. 主テキストの番号，順序，仮定，結論，記号を一覧化する。
4. 対応するユーザー作成ノートがあるか確認する。
5. 手書きノートがある場合は，赤字，青字，黄色，図式，中間主張，欄外メモを抜き出す。
6. 矢印の先，矢印の脇，引き出し線の先に数学的主張・同値性・補題・注意が書かれている場合は，単なる装飾や参照印ではなく，本文補足候補として記録する。
7. 手書きノートに参考文献名，著者名，URL，arXiv 番号，Stacks tag，MathOverflow/Math StackExchange 番号，定理名・補題名がある場合は，本文へ入れる前に台帳と照合する。
8. URL や文献名がなくても，色付き矢印，引き出し線，丸囲み，下線，括弧などが根拠候補を指している場合は，始点・終点，色，周辺語句を記録する。
9. 証明を補完する場合は，主テキストと手書きノートから，帰着，構成，有限段階，単射性，全射性，極限への移行，既知補題の適用など，証明の役割が変わる区切りを作業用の証明段階として読み取る。これは完成版に `Step 1` などの見出しを機械的に置く指示ではない。
10. 各証明段階の内部で使う具体的な確認を，必要なら `claim` または局所的な `lem` の候補にする。
11. 地の文・注意・例から切り出す主張と，証明内で証明段階を支える局所主張・補題を別々に記録する。
12. 本文案を書く前に，補うべき定義，文脈，証明段階，既出参照，参考文献根拠を短く列挙する。
13. 証明内で使う知識が，既に本文，注意，定義，命題，補題，定理，主張，例で示されていないか確認する。
14. 既出なら，同じ内容を再度 `claim` や `lem` として証明せず，既存箇所へ参照で飛ばす。
15. 標準補題や強い道具を使う場合は，主テキスト，手書きノート，対象参考文献の文脈に属するかを確認する。文脈から離れた道具は，単に結論が出るという理由だけで持ち込まない。
16. 標準補題を使う前に，定義，基底表示，剰余環，生成元，付値評価，直和分解などから素朴に示せないか確認する。
17. 地の文，注意，例の中で実質的な数学的主張を使う箇所を拾う。証明や確認が必要で，既出結果として参照できないなら，`claim`，`lem`，`prop`，`dfn`，`recall`，`localdfn` など既存環境で切り出す。
18. 証明内で `claim` または局所的な `lem` を使う場合は，直前に「まず次の主張を示す。」「次の補題を示す。」型の文を置く。
19. 証明内の `claim` は文脈に依存してよいが，複数の対象や場合分けがある場合は対象を明示する。証明内の局所的な `lem` は，独立した命題として読めるように仮定と結論を環境内に書く。
20. 補助主張または局所補題の証明が終わる箇所には，最後の文の直後に `\localqed` を置く。音符だけを独立行に置かない。
21. 本文では作業資料名を出さず，自然な教科書本文に統合する。
22. TeX 本文を変更した場合は，`latexmk -g -dvi -interaction=nonstopmode main_final.tex` でビルドする。
23. 完成版 PDF と必要な参考文献を対象に独立査読を行う。この段階では主テキスト，手書きノート，改訂履歴を見ない。
24. ユーザー判断が必要な案は `REVISION_CANDIDATES.md` に入れる。
25. `git diff` を確認して報告する。

## 3. 停止条件

次のどれかに当たる場合は，本文へ曖昧に混ぜない。

1. 仮定が現在の状況で満たされるか確認できない。
2. 参考文献の定理番号・命題番号・ページが特定できない。
3. 手書きノートの判読が怪しい。
4. 用語が `TERM_DICTIONARY.md` の未承認語に当たる。
5. 主テキスト，手書きノート，参考文献が衝突している。
6. 定義，well-defined 性，連続性，同変性，単射性，全射性，同型性の確認が抜けている。
7. 手書きノートの根拠候補や色付き矢印を特定できない。

停止した場合は，本文変更ではなく，候補，未解決点，確認先を報告する。

## 4. 報告形式

作業後は次を簡潔に報告する。

1. 対象範囲
2. 変更ファイル
3. 主な変更
4. 実行した確認
5. ビルド結果
6. 未解決点
7. commit するなら推奨 commit 単位
